## Build & run

- WPF, .NET 8, Windows-only, x64-only (no AnyCPU). Configurations: `Debug|x64`, `Release|x64`.
- Build: `dotnet build src/OES_Leak_Monitor/OES_Leak_Monitor.csproj -c Debug` (or open `OES_Leak_Monitor.sln` in Visual Studio 2022).
- Run: `dotnet run --project src/OES_Leak_Monitor/OES_Leak_Monitor.csproj`.
- No test project exists yet.
- **From the WSL shell there is no native `dotnet`** — this is a Windows/WPF build. Invoke the Windows SDK explicitly, e.g. `"/mnt/c/Program Files/dotnet/dotnet.exe" build src/OES_Leak_Monitor/OES_Leak_Monitor.csproj -c Debug` (same for `publish`). The bare `dotnet …` commands above assume a Windows terminal / Visual Studio.

### Manual verification (no hardware required)

There is no automated test suite, so changes are validated by running the app. With no spectrometer attached, Connect falls back to **test mode** (`ConnectStandaloneAsync`), which streams synthetic spectra — enough to exercise the full UI without hardware. A connect that silently lands in test mode when hardware *should* be present usually means the native DLLs didn't reach the output root (see the native-DLL note above). To smoke-test a change:

- Launch, then **Connect** → **Start** in the Monitor tab's `DevicePanel`; confirm spectra stream and the Leak Monitor / Wavelength-Trend charts update.
- The Leak Monitor tab shows a **"test mode"** note while synthetic; alarms are suppressed there by default (`LeakMonitorSettings.SuppressAlarmsInTestMode`), so to exercise alarm transitions either flip that flag or feed real/abnormal data.
- The intensity logger only writes CSVs while the selected wavelength's intensity is **above the trigger threshold** — lower the threshold in the `LoggerPanel` if you need test-mode spectra to cross it and open a save session (the Ratio CSV opens/closes in lockstep).
- Recordings / Ratio Review tabs read back the CSVs written above; use them to confirm a logging change round-trips to disk and re-parses.

### Standalone publish (no .NET install required on the target PC)

`Properties/PublishProfiles/SelfContained-win-x64.pubxml` produces a single self-contained `.exe` (the .NET 8 runtime is bundled in). Three ways to run it: double-click `publish.cmd` in the repo root; `dotnet publish src/OES_Leak_Monitor/OES_Leak_Monitor.csproj -p:PublishProfile=SelfContained-win-x64`; or in Visual Studio right-click the project → Publish → that profile. The `/publish` Claude slash command does the same. Output: `src/OES_Leak_Monitor/bin/Publish/win-x64/`.

The `.exe` bundles all managed code plus the .NET 8 runtime, but native DLLs are not embedded — the output folder also holds the WPF native DLLs (`wpfgfx_cor3.dll`, `PresentationNative_cor3.dll`, `D3DCompiler_47_cor3.dll`, `PenImc_cor3.dll`, `vcruntime140_cor3.dll`) and the OES native DLLs (`UserApplication.dll`, `SiUSBXp.dll`). Ship the whole folder together — those DLLs are loose next to the `.exe` on purpose (the `DllResolver` only searches the app base directory; see the native-DLL note below). Do not enable `IncludeNativeLibrariesForSelfExtract` — it would bundle the OES DLLs into the exe and break hardware connect.

### NuGet feed prerequisite

`nuget.config` pins a local feed at `C:\Users\infor\source\repos\Ray1962\LocalPackages` and clears default sources before adding nuget.org. Restore will fail if the three `Aqst.*` nupkgs (`Aqst.OesApp.Core`, `Aqst.OesApp.Wpf`, `Aqst.OesSpectrometer`) are not present there. ### Native DLL flattening (do not remove)

`Aqst.OesSpectrometer` ships `UserApplication.dll` and `SiUSBXp.dll` under `runtimes\win-x64\native\`, but the package's `DllResolver` only searches the app base directory. The `FlattenOesNativeDlls` MSBuild target in `OES_Leak_Monitor.csproj` copies those two DLLs into `$(OutDir)` (and `$(PublishDir)`) after Build/Publish. If hardware connect silently falls back to test mode, check that those files reached the output root.

## Architecture

This is a **single-OES** leak-monitoring fork of a previously dual-OES codebase. Most of the UI shell, device control, logging, access control, and plot/configuration panels live in the `Aqst.OesApp.*` NuGets — the project here is mostly composition + a custom Recordings tab.

### What comes from `Aqst.OesApp.Core` / `Aqst.OesApp.Wpf` (referenced, not defined here)

- `DeviceViewModel`, `LoggerViewModel`, `LogViewerViewModel`, `SystemLogger`, `DualIntensityLogger`, `IntensityCsvWriter`
- `AccessControlService`, `UserRole` (Guest < Operator < Engineer < Admin), `LoginDialog`, `UserManagementDialog`, `AccessControlConfig`
- `OesAppPaths` — resolves per-user AppData subfolders (`Config`, `Log`, `Data`) under the app folder name
- `RelayCommand`, `DeviceSettings`, `LoggerSettings`, `LogSeverity`
- WPF user controls bound in `MainWindow.xaml`: `DevicePanel`, `ConfigurationPanel`, `LoggerPanel`, `LogViewerPanel`
- Global `using Aqst.OesApp.Core;` and `using Aqst.OesApp.Wpf;` are injected via the csproj, so these types appear unqualified throughout.

Treat that NuGet surface as the framework. When something looks "missing," it is almost certainly defined in those packages, not here.

### What is defined in this project

- `MainViewModel` — wires a single `DeviceViewModel` (the `DeviceProfiles` array has one entry tagged `"OES1"`), the `DualIntensityLogger`, the `RecordingsViewModel`, and the toolbar role badge. Per-device Connect/Disconnect/Start/Stop live in the embedded `DevicePanel` header (Monitor tab); the LoggerPanel exposes Start Save / Stop Save. The top toolbar holds only the title, status message, and the user/role controls — there is no longer a "Both" command that bundles device + logger.
- `AppSettings` + `SettingsService` — `settings.json` in `OesAppPaths.ConfigDirectory`. `AppSettings.OnDeserialized` migrates the legacy `device1`/`device2` keys into the `devices` list, and pads the list to at least 2 entries (legacy schema lock — do not shrink to 1).
- `Recording` / `RecordingGroup` — parse CSVs produced by `IntensityCsvWriter` under `{baseDir}\YYYYMM\DD\{prefix}_{tag}_{MMddHHmmss}[_N].csv`. The group key is `{prefix}|{sessionStart}|{rotation}` (dual-OES file layout preserved) but only the `"OES1"`-tagged file populates the group; any historical `"OES2"` siblings on disk are ignored at scan time. Year is recovered from the parent folder.
- `RecordingsViewModel` + `RecordingsPanel` — Recordings tab: scans the data folder, builds OxyPlot line / heatmap / frame-spectrum views, supports two-row compare mode (two sessions overlaid — orthogonal to device count), PNG export, clipboard copy, notes, search.
- `RatioReviewViewModel` + `RatioReviewPanel` + `RatioCsvReader` — Ratio Review tab: scans the data folder for the `{prefix}_Ratio_*.csv` files `RatioCsvLogger` writes, and plots the selected file's per-ratio trend in one of three modes — % of baseline (with the 100/120/150 % threshold lines), raw ratio, or **leak rate** (the calibrated `Q̂` in mbar·L/s with a translucent ±1σ band) — with the composite leak state drawn as translucent Warning/Alarm bands behind the series. The ratio CSV carries trailing `LeakRate,LeakRateSigma` columns (P3); `RatioCsvReader` locates columns by header name so pre-calibration files (without those columns) still parse, exposing `RatioTrendData.HasLeakRate`. Reuses `Recording.TryParse` for the shared filename scheme, filtering on the `"Ratio"` tag. PNG export, clipboard copy, search.
- `RatioSetupViewModel` + `RatioEditViewModel` + `RatioSetupPanel` — Ratio Setup tab: a staged editor for the species-ratio configuration. Up to `RatioSetupViewModel.MaxRatios` (5) ratios; each picks a signal and a reference line (species + wavelength) from the species-grouped `SpectralLineCatalog`, plus per-ratio extraction mode and alarm thresholds. Save persists into `LeakMonitorSettings.Ratios` in `settings.json` (same file as the Configuration tab), gated to Engineer+. Edits are **staged** — they take effect only when OES acquisition restarts: `MainViewModel` watches `DeviceViewModel.IsAcquiring` and, on a Stop→Start transition, calls `LeakMonitorEngine.ReloadRatios()`, which rebuilds the monitors and raises `RatiosReloaded` (the Leak Monitor tab rebuilds its rows in response). `RatioEditViewModel` reuses an untouched line's original `LineRegion` so editing one ratio never re-labels another. The Leak Monitor tab is therefore display-only for ratio config (no live reference/enable controls).
- `WavelengthTrendViewModel` + `WavelengthTrendPanel` — the live intensity time-trend chart below the `DevicePanel` on the Monitor tab. It subscribes to `DeviceViewModel.SpectrumAvailable` and plots one line per tracked wavelength: the trigger ("threshold") wavelength configured in the `LoggerPanel` (the "selected" wavelength — not a separate setting), plus the first `MaxMonitoredWavelengths` (5) entries of that panel's monitored-wavelength list (the wavelengths logged into the intensity CSV). Each line is the peak intensity within ±0.5 nm of its wavelength; a dashed reference line marks the save-start threshold intensity. `MainViewModel` seeds it from `LoggerViewModel.ToSettings()` and re-points it (`Configure`) on every `ApplyAll`; any change to the tracked-wavelength set rebuilds and clears the trends. A **Normalize** checkbox divides every line by its value at the moment the box is ticked (each `Track` keeps its raw points plus a `Baseline` captured on enable), so all trends snap to 1.0 and then show divergence; the threshold line is hidden while normalized. 30-min retention, follow-while-zoomed, Zoom All — same chart conventions as the Leak Monitor trend.
- `LeakRateEstimator` + `LeakCalibrationViewModel` + `LeakCalibrationPanel` + `CalibrationPointDialog` — Leak Calibration tab: a staged wizard that turns the qualitative ratio rise into a **quantitative leak rate** (mbar·L/s). The operator picks a baseline (Golden Run), then captures one calibration point per calibrated-leak element: `LeakMonitorEngine.BeginCalibrationPointCapture(leakRate, label, seconds)` averages each ratio's fractional rise `x = rawRatio/baselineMean − 1` over a window (reusing the Golden Run capture machinery) and raises `CalibrationPointCaptured`. Save calls `LeakRateEstimator.FitAll` — a through-origin weighted least-squares sensitivity `x ≈ s·Q` per ratio — and persists a `LeakCalibration` (points + per-ratio `RatioSensitivity`, bound to the baseline and reference labels) into `LeakMonitorSettings.Calibrations` in `settings.json`, gated to Engineer+. At runtime (P2) the engine holds a `LeakRateEstimator` built from the active calibration and computes a fused estimate every frame into `LeakMonitorSnapshot.LeakRate`: each ratio's `xᵢ` comes from its % -of-baseline and `σ_xᵢ` from a per-ratio EWMA variance (`RatioMonitor._emaVar`); `Estimate` inverts each to `Qᵢ = xᵢ/sᵢ` and fuses them by inverse variance into `Q̂ ± σ_Q` with a cross-ratio consistency confidence and an out-of-calibrated-range flag. The Leak Monitor panel shows it as a readout under the status banner ("Leak rate ≈ Q̂ ± σ · NN% confidence", or `extrapolated` / `not calibrated`). Saving a calibration calls `engine.ReloadCalibration()` so it activates immediately (no acquisition restart, unlike a ratio-set change). Pure math core (`LeakRateEstimator`) has no engine/UI deps. **The calibrated leak must be air** (or match the leak being detected) — an inert leak won't raise O/OH/NO. The estimate is also logged to the Ratio CSV (`LeakRate,LeakRateSigma` columns) and replayable in Ratio Review's "Leak rate" mode (P3). A calibration only applies while the active Golden Run matches the baseline it was captured against (`LeakMonitorSnapshot.CalibrationStatus`: `NotCalibrated` / `Active` / `BaselineMismatch`); switching baselines suspends estimation rather than silently mis-scaling it, and status transitions are logged (`LeakCalibrationActive` / `LeakCalibrationSuspended` / `LeakCalibrationCleared`). Every baseline change routes through `ApplyGoldenRun` → `BuildEstimator` to re-evaluate validity (P4). Full design + math: `docs/leak-rate-calibration.md` (P0–P4 complete: model, estimator, wizard, runtime display, CSV logging/review, validity gating).
- **Leak Monitor** — an actinometry-based air-leak detector on its own tab. `LineIntensityExtractor` + `LineRegion` pull baseline-subtracted line intensities out of each `SpectrumSample` (`LineRegion.PeakSearchHalfWidthNm` re-centers the window on the actual peak each frame, so wavelength-calibration drift or a shifted band head doesn't corrupt the reading); `LeakMonitorEngine` divides signal lines by a reference line into ratios (`R_O`, `R_OH`, `R_NO`, `R_Ar`) — the reference defaults to N₂ 337.1 nm but is switchable per ratio from `ReferenceLineCatalog` (N₂ 337.1, N₂ 662.4, Hα 656.3, Ar 750.4, Ar 811.5); a Golden Run baseline only applies while the ratio's reference still matches the one it was captured against. The engine runs a per-ratio `RatioMonitor` (EMA + two-level threshold state machine + sustained-confirmation + latched alarm), and computes a composite `LeakAlarmLevel`. A "Golden Run" captures the leak-free baseline (mean/σ per ratio) and is persisted to `settings.json` the moment it completes. `LeakMonitorViewModel` / `LeakMonitorPanel` show the per-ratio rows, the % -of-baseline trend chart, and the capture / acknowledge commands. `SpectralLineCatalog` is the static reference line table. Engine alarm transitions and Golden Run captures bridge into `SystemLogger`. `RatioCsvLogger` writes the ratio trend (raw ratio + % of baseline per ratio) to a `{prefix}_Ratio_{MMddHHmmss}.csv` sibling of each intensity-logger save session — same `DataDirectory\YYYYMM\DD` folder, opened/closed in lockstep with the threshold logger so rows are only written while plasma intensity is above the trigger threshold. All of this lives in this project, not the NuGets.

### Conventions worth knowing

- The single device keeps the tag `"OES1"` (not `"OES"`) so the existing recording file-pairing infrastructure, filename scan, and `RecordingGroup` logic keep working unchanged. Don't rename it.
- The Configuration tab is gated to Engineer+; window close is blocked for Guest. Both gates are enforced in `MainWindow.xaml.cs` (`MainTabControl_SelectionChanged`, `MainWindow_Closing`).
- Sample flow: each `DeviceViewModel.SpectrumAvailable` event is forwarded to `DualIntensityLogger.ProcessSample(slot, sample)`. The logger's threshold state machine decides when to open/close CSV writers; lifecycle events (`StateChanged`, `ErrorOccurred`, `FilesChanged`) are bridged into `SystemLogger` for the audit log.
- `MainViewModel.ConnectBothAsync` calls `OesDiscovery.OpenAllDevices()` once, then hands one handle to each slot via `AttachAsync`; slots with `ForceTestMode=true` or that run out of handles fall back to `ConnectStandaloneAsync`. Unused handles are explicitly closed.
- Settings are saved atomically (temp file + `File.Move(..., overwrite:true)`). The `AccessControl` user list is re-read from disk on each persist so user-management edits don't clobber unsaved Configuration-tab edits, and vice versa.
